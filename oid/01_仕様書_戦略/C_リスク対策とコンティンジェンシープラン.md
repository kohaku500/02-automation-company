# C. リスク対策＆コンティンジェンシープラン

## 概要
Month 1（6月）から Month 3（8月）の運用期間中に、予期しない事態が発生した場合の対応フローを定義します。  
各リスクに対して「検知方法」「初期対応」「対応フロー」「エスカレーション」を明記し、迅速な判断・行動を可能にします。

---

## リスク評価フレームワーク

### リスク重要度の定義
```
■ レベル1（Critical）：ビジネスが停止する可能性
  例：システムダウン、大規模セキュリティ侵害、主要顧客離脱

■ レベル2（High）：ビジネス継続は可能だが重大な影響
  例：営業効果が目標比30%未満、サポート対応困難、大幅な予算超過

■ レベル3（Medium）：運用に支障あるが対応時間の余裕あり
  例：機能不具合、顧客数社の不満、軽微なセキュリティ警告

■ レベル4（Low）：長期的な改善対象だが緊急性なし
  例：UI改善要望、非コア機能の不具合、マイナー費用超過
```

## リスク 1: ユーザー獲得目標の大幅未達

### リスク概要
- **事象**：6月の新規サインアップ数が目標比30%以下
- **重要度**：Level 2（High）
- **発生確度**：中程度
- **影響度**：月次売上、キャッシュフロー、チーム士気

### 検知方法
```
■ 検知タイミング
  - 毎日の日次レポート（登録数トレンド）
  - 週次レポート（週平均登録数 vs 目標達成ペース）
  - 営業パイプライン分析（商談予定数の確認）

■ 検知指標
  - 週単位で「累計登録数」が目標の50%未満
  - 営業接触数が計画比70%未満
  - メール開封率 / クリック率が業界平均比30%以下
```

### 対応フロー

#### 対応1-1: 事実確認と原因分析（24時間以内）
```
チェック項目：
□ 過去3週間の登録数トレンドグラフ作成
□ セグメント別に登録数を分解（セグメント1: X名 / セグメント2: X名）
□ 営業活動ログの詳細確認
  - 接触数は計画通りか？
  - 各営業チャネル（Email / 直営業 / 紹介）別の効果測定
□ 初期顧客候補リストに「対応済み / 未対応」を振り分け
□ Web トラフィック分析
  - サイト訪問者数は充分か？
  - 登録ページへの到達率は？
```

成果物：`原因分析レポート.md`

#### 対応1-2: 短期施策の実行（3～7日）
```
■営業チャネル別の緊急強化
  □ Email営業：配信頻度を週2→週3回に増加、テンプレート3種類を A/B テスト
  □ 直営業：手作業による電話営業・オンライン面談を追加（週20件以上）
  □ 紹介：既存顧客へ「紹介キャンペーン」（割引or報酬）を提案
  □ SNS / Web 広告：広告費を [¥X] → [¥Y] に増額、ターゲティング調整

■既存候補リスト再活性化
  □ 未対応の候補企業30社を優先リスト化
  □ 営業アプローチを段階化（初期接触→2回目接触→トライアル誘導）
  □ 返信のない候補に対して「異なる接触手段」を試す（Email→電話→LinkedIn等）

■営業・マーケティング資料の緊急改善
  □ 営業資料の「課題解決」フォーカスを強化
  □ 初期接触メールの件名・本文を A/B テスト
  □ 顧客の「購買障壁」（予算 / 導入期間 / 既存システムとの連携）を把握
```

成果物：`短期営業強化プラン.md`

#### 対応1-3: 中期的な施策検討（1～2週間）
```
■営業戦略の再検討
  □ ターゲットセグメントの見直し
    - 現在のセグメント定義では「レスポンス」が低いのか？
    - より購買意欲の高い「新規セグメント」に軸足を移すべきか？
  
  □ 価格設定の見直し
    - 無償トライアル期間を延長する（30日→60日）？
    - 初期価格を引き下げる（¥X→¥Y）？
    - バンドル商品を作成する？
  
  □ 営業体制の強化
    - 外部営業代理店との契約を検討するか？
    - パートタイム営業担当者の採用を検討するか？
    - マーケティング業務を外注化するか？

■パートナーシップ・提携開拓
  □ 相互補完的なサービス提供企業とのパートナーシップ
  □ 業界団体・協会を通じた認知拡大
  □ インフルエンサー / KOL との協業

成果物：`中期営業戦略見直し案.md`
```

#### 対応1-4: エスカレーション（判断フロー）
```
■以下の条件で「戦略転換会議」を開催
  - 対応1-2実施後も改善傾向が見られない場合
  - または目標未達の確定が「月中盤」に判明した場合

■経営層への報告・意思決定
  □ 6月目標達成の「可能性有無」を正式判定
  □ 必要な予算追加 / リソース追加を確認
  □ 目標値の現実的な下方修正の承認
  □ 7月以降のビジネス方針（スケール継続 or ピボット）の決定
```

## リスク 2: ユーザー定着率の急速な低下（高チャーン）

### リスク概要
- **事象**：初期ユーザーが想定外のペースで離脱（7日継続率が50%未満）
- **重要度**：Level 2（High）
- **発生確度**：中程度
- **影響度**：ユーザーベース喪失、ネガティブ口コミ拡散

### 検知方法
```
■検知タイミング
  - 毎日の日次レポート（アクティブユーザー数トレンド）
  - ユーザーの「最後のログイン日」の自動監視
  - チャーンスキャン（7日間ログインなし = チャーン）

■検知指標
  - 7日継続率が40%未満
  - サポートチケットに「辞めます」「使いにくい」などの内容が増加
  - 顧客インタビューで「期待と異なる」という声が複数
```

### 対応フロー

#### 対応2-1: チャーン要因の分析（24～48時間以内）
```
チェック項目：
□ ユーザー行動ログの詳細分析
  - ログインパターンを可視化
  - 「どのステップで止まるのか」を特定（登録直後 / 初期操作 / X日目など）
  - 利用する機能 vs 利用されない機能を分析

□ チャーンしたユーザー（離脱者）への直接ヒアリング
  - メール / 電話で「なぜやめたのか？」を質問
  - 最低3～5人のチャーン者からフィードバック取得

□ 継続ユーザーとの比較分析
  - 「継続している顧客」と「チャーンした顧客」の特性の違い
  - ユーザーセグメント別のチャーン率（セグメント1: XX% / セグメント2: XX%）

成果物：`チャーン原因分析レポート.md`
```

#### 対応2-2: 緊急リテンション施策（3～5日）
```
■即座に実施できるオンボーディング改善
  □ サインアップ直後の「ウェルカムメール」を改善
    - より簡潔で「最初の1ステップ」に焦点
    - 動画チュートリアルへのリンク追加
  
  □ 新規ユーザーへの主動的サポート
    - 登録から48時間以内に「使えていますか？」メール送付
    - 3日目に「つまずいていないか？」電話チェック
    - 1週間目に「目標達成できた？」ヒアリング

■既にチャーンしたユーザーのリカバリー
  □ 「サービスを中止した」ユーザーへの特別オファー
    - 「戻ってきてくれたら、サポートチーム専任つけます」
    - または「1ヶ月無料延長」「割引プラン」の提案
  
  □ 原因が「使い方がわかりにくかった」の場合
    - 1対1サポート（有料 / 無償 どちらでも）を提案

成果物：`リテンション改善プラン.md`
```

#### 対応2-3: プロダクト改善（1～2週間）
```
■チャーン原因がプロダクル側にある場合の改善案
  □ ユーザビリティ改善
    - 初期操作画面の UI/UX 改善
    - FAQ 充実、インアプリチュートリアル追加
  
  □ 期待値設定の改善
    - ランディングページ / トライアル説明を修正
    - 「このツールで達成できること / できないこと」を明確化
  
  □ 機能優先度の見直し
    - 「ユーザーが実際に使う機能」と「企画時の想定」にズレがないか確認
    - リリース優先度を見直す

成果物：`プロダクト改善実行計画.md`
```

#### 対応2-4: エスカレーション（判断フロー）
```
■以下の場合は経営層判断が必要
  - 継続率が20%未満（ほとんどのユーザーがチャーン）
  - チャーン理由が「根本的な機能不足」である場合

■意思決定ポイント
  □ 現在のビジネスモデル / プロダクト戦略が適切か再評価
  □ 大規模なプロダクト改造が必要な場合の工数・予算確認
  □ ターゲット顧客の再定義が必要か判定
```

## リスク 3: システム障害・セキュリティインシデント

### リスク概要
- **事象**：システムダウン / データ漏洩 / 不正アクセス / 大規模バグ
- **重要度**：Level 1（Critical）
- **発生確度**：低～中程度（準備状況に依存）
- **影響度**：ビジネス停止、信用喪失、法的責任

### 検知方法
```
■自動監視
  - サーバー稼働状況の常時監視（Uptime 監視ツール）
  - エラーログの自動アラート
  - 異常アクセスパターンの検知

■ユーザーからの報告
  - サポートチャネルへの「使えない」「エラーが出た」報告
  - SNS での「サービス障害」言及の検知

■検知指標
  - API エラー率が1%を超える
  - 複数ユーザーから同時に不具合報告
  - ダッシュボード上での異常な数値
```

### 対応フロー

#### 対応3-1: 初期対応（即時～15分以内）
```
■事象判定
  □ システムダウンか、軽微なバグか判定
  □ 影響範囲を確認（全ユーザー / 特定機能 / 特定セグメント）
  □ データ漏洩の可能性の有無を初期評価

■緊急連絡体制発動
  □ エンジニア全員に緊急アラート（Slack / 電話 / その他）
  □ 経営層・法務への報告（必要に応じて）
  □ 外部セキュリティ相談者への連絡（必要に応じて）

成果物：`インシデント報告書（初期版）`
```

#### 対応3-2: 対応フロー（15分～2時間以内）
```
■一時対応
  □ 問題が特定機能に限定される場合
    - その機能をオフラインにする（ユーザーへの告知あり）
  
  □ システム全体が落ちている場合
    - バックアップシステムへのフェイルオーバー実行
    - または段階的な復旧スケジュール公開

■ユーザー・顧客への周知
  □ ステータスページ（ダッシュボード / Web）で状況を常時更新
  □ 主要顧客への個別メール / 電話連絡
  □ SNS でも事実確認と対応状況を発表

■根本原因の調査と復旧
  □ ログ / コード / インフラを確認
  □ 再発防止措置を講じながら復旧
  □ 復旧が完了するまで、顧客への定期更新を継続

成果物：`障害対応ログ`
```

#### 対応3-3: 事後対応（復旧から24～72時間）
```
■ユーザーへの謝罪・補償
  □ 謝罪メール / アナウンス
  □ 補償措置の検討（無料期間延長 / 割引 など）

■根本原因分析と予防策
  □ なぜこの障害が発生したのか、技術的に分析
  □ インシデント報告書（最終版）を作成
  □ 再発防止策（テスト強化 / インフラ改善 / 監視強化 など）を決定

■法的・コンプライアンス対応
  □ データ漏洩がある場合：
    - 監督官庁への報告（法律で定められている場合）
    - 影響を受けたユーザーへの通知
  
  □ 保険・法務相談（必要に応じて）

成果物：`インシデント後報告書・改善計画`
```

#### 対応3-4: エスカレーション（判断フロー）
```
■データ漏洩の可能性がある場合
  - 外部法務 / セキュリティ専門家への即座の相談
  - 監督官庁への報告義務の確認
  - 事後対応の全体スケジュール決定

■経営層への重要性報告
  - 技術的なリスク内容を「経営インパクト」として翻訳して報告
  - 再発防止の予算確保の是非判定
```

## リスク 4: 主要顧客の離脱・大型案件のキャンセル

### リスク概要
- **事象**：重要顧客（月額 ¥X以上）が急に解約を申し出る
- **重要度**：Level 2（High）
- **発生確度**：中程度
- **影響度**：月間売上が大幅減少、キャッシュフロー悪化

### 検知方法
```
■検知タイミング
  - サポートチケット / ユーザーからの「解約相談」
  - 顧客とのコミュニケーション頻度の急な低下
  - 月次チェックイン時の顧客の「トーン」の変化

■検知指標
  - 重要顧客からの「解約予定」通知
  - 3週間以上、重要顧客からの連絡が途絶える
  - サポート満足度スコアの急低下
```

### 対応フロー

#### 対応4-1: 事象確認・チャーン理由の把握（24時間以内）
```
チェック項目：
□ 顧客との即座の電話 / 面談で解約理由をヒアリング
□ 解約理由を分類
  - 機能不足 / 使いにくさ
  - 価格が合わない
  - 経営判断（予算削減など）
  - 競合乗り換え
  - サービス品質への不満

□ 顧客の「踏みとどまる可能性」を評価
  - 理由が「一時的な予算問題」なら対応可能性高
  - 理由が「根本的な機能不足」なら対応困難の可能性

成果物：`顧客離脱ヒアリング記録`
```

#### 対応4-2: リテンション交渉（24～72時間以内）
```
■顧客の「踏みとどまる可能性」が高い場合
  □ 解約の「猶予期間」の延長交渉
    - 「30日待ってください、改善案を提案します」
  
  □ 価格交渉 / 割引オファー
    - 月額を [¥X] → [¥Y] に低減（期間限定）
  
  □ サービス内容の改善・カスタマイズ
    - 顧客の「不足していた機能」を優先実装する約束
    - 専任サポーターの配置

■顧客の「踏みとどまる可能性」が低い場合
  □ 潔い対応
    - 解約手続きをスムーズに進める
    - 「今後もし状況が変わったら戻ってきてください」という関係保持

成果物：`顧客リテンション交渉レコード`
```

#### 対応4-3: 根本原因への改善（1～2週間）
```
■「失いかけた顧客」から学ぶ
  □ なぜこの顧客の不満に早期に気づけなかったのか
  □ オンボーディング / サポート / コミュニケーション計画を見直し
  □ 「顧客の成功」を測定する KPI を設計

■他の重要顧客への予防的対応
  □ 月1回の定期チェックイン（「成功していますか？」確認）
  □ サポート満足度スコアの定期採集
  □ 「隠れた不満」を早期発見する仕組み

成果物：`顧客サクセス改善計画`
```

## リスク 5: 営業資源・人員不足

### リスク概要
- **事象**：営業 / カスタマーサポート / エンジニア が計画した人数揃わない、またはキーパーソンが離職
- **重要度**：Level 2（High）
- **発生確度**：中程度
- **影響度**：営業活動停滞、サポート品質低下、プロダクト開発遅延

### 検知方法
```
■検知タイミング
  - スタッフの採用公募に「応募が少ない」「良い候補がいない」
  - 既存スタッフからの「退職予定」通知
  - タスク負荷の過度な集中化（1人への依存度が高い）

■検知指標
  - 営業接触数が計画比70%未満（人員不足が原因の場合）
  - サポートチケット対応時間が SLA を超える傾向
  - バグ修正 / 機能開発のスケジュール遅延
```

### 対応フロー

#### 対応5-1: リソース再配置・タスク優先化（48時間以内）
```
チェック項目：
□ 現在の各チームの「実際のキャパシティ」を把握
  - 営業：実際に何件/週の接触ができるか
  - サポート：実際に何件/日対応できるか
  - エンジニア：実際に何時間/週開発に充てられるか

□ 優先度の最優先化（全てのタスクはできない前提で）
  - 売上直結タスク を優先
  - 顧客離脱防止 を優先
  - その他は後回し or 削除

□ 内部プロセスの効率化
  - 「これは本当に必要か？」という問い直し
  - 手作業を簡略化（テンプレート化 / 自動化）

成果物：`タスク優先度再編成リスト`
```

#### 対応5-2: 緊急採用 / 外注化（1～2週間）
```
■即座の採用活動
  □ 派遣社員 / インターン / フリーランスの採用
    - 採用スピードを優先（「完全な適合」は後で調整）
    - 「営業代行」「サポート代行」など部分的な外注でもOK
  
  □ 采用条件の緩和
    - 経験年数 / スキル要件を下げる
    - 試験期間（短期契約）でスタート

■既存メンバーへのサポート
  □ 過度な負荷を軽減する（残業 / 休日出勤を避ける）
  □ 「新メンバーのオンボーディングは既存メンバーが担当」を避ける

成果物：`緊急採用計画書`、`外注先リスト`
```

#### 対応5-3: キーパーソン離職への備え（随時）
```
■もし「重要な役割を担う人が辞める」と告げられた場合
  □ 即座に「その人の仕事内容」をドキュメント化
  □ 引き継ぎ期間の確保（最低2週間）
  □ 後任候補の準備 / トレーニング

■平時からの予防
  □ 「キーパーソンへの依存度を下げる」設計
    - プロセスの標準化（「Aさんにしかできない」を避ける）
    - ナレッジ共有の仕組み（ドキュメント / ウィキ）
```

## リスク 6: 予算大幅超過 / キャッシュフロー枯渇

### リスク概要
- **事象**：サーバー費用 / 広告費 / 人件費などが予算を大幅に超過、または赤字が想定以上
- **重要度**：Level 2（High）
- **発生確度**：中程度（準備段階での見積もり誤差に依存）
- **影響度**：資金繰り困難、事業継続リスク

### 検知方法
```
■検知タイミング
  - 月次経営会議での「実績 vs 予算」比較
  - 銀行口座残高の急減少

■検知指標
  - 実績が予算比120%を超える（3項目以上）
  - キャッシュ残高が「6ヶ月のランウェイ」を割り込む
  - 追加の外部資金調達ができない状況
```

### 対応フロー

#### 対応6-1: 詳細な支出分析（24～48時間以内）
```
チェック項目：
□ 支出項目の詳細分解
  - サーバー / インフラ [¥X] → 何が高いのか？
  - 広告費 [¥X] → ROI がどうなっているか？
  - 人件費 [¥X] → 人数は想定通りか？
  - その他 [¥X]

□ 各支出項目の「必要度」評価
  - 「これは絶対に必要か？」「後で削減できるか？」
  - 「短期 / 中期 / 長期」で段階的な削減が可能か？

成果物：`支出詳細分析表`
```

#### 対応6-2: 緊急削減措置（即座～1週間）
```
■即座に実行できる削減
  □ 広告費の削減 / 一時停止
    - ROI が低い広告を即座に停止
    - 予算を再配分（効果的なチャネルに集中）
  
  □ サーバーコストの最適化
    - 不要なリソース（データベース / ストレージ）の削除
    - より安い料金プランへの切り替え
  
  □ オフィス / インフラの削減
    - サテライトオフィス解約
    - ライセンス数削減（使用していないツール）

■遅延可能なプロジェクトの延期
  □ 新機能開発 → 優先度1のみ実施、それ以外は延期
  □ マーケティングキャンペーン → 計画再検討

成果物：`緊急削減計画書`
```

#### 対応6-3: 収入最大化施策（1～2週間）
```
■売上向上施策の加速
  □ 営業 / マーケティングの集中投下（効果的なチャネルのみ）
  □ 高単価顧客の獲得への注力
  □ 既存顧客の「アップセル」（より高いプランへの引き上げ）

■価格設定の再検討
  □ 無料トライアルを有料オプションに変更
  □ 価格体系の見直し（段階的値上げ）

■外部資金調達の検討
  □ 既存投資家からの追加融資
  □ 銀行融資 / ローンの申請
  □ 別事業との統合 / 提携による収入源開拓
```

## リスク 7: 競合・参入障壁の脅威

### リスク概要
- **事象**：大手企業が同じジャンルに参入、または新興競合が出現し、市場シェア / 価格競争力を失う
- **重要度**：Level 2（High）
- **発生確度**：中程度
- **影響度**：市場シェア喪失、価格下げ圧力、ビジネスモデル崩壊の可能性

### 検知方法
```
■検知タイミング
  - 業界ニュース / 競合企業の動向監視
  - 顧客からの「競合のサービスも検討している」という言及
  - 営業時の「競合比較」質問の増加

■検知指標
  - 競合の「同等機能」が「より低価格」で提供されている
  - 市場規模が競合多数で「奪い合い」になっている
  - 営業成約率が低下（「競合と比較中です」という返答が増える）
```

### 対応フロー

#### 対応7-1: 競合分析（1週間以内）
```
チェック項目：
□ 競合企業のリストアップ（直接競合 5社 + 間接競合 10社）
□ 競合との「機能比較表」作成
  - 自社の強み / 弱み を明確化
  - 「自社にしかない機能」は何か？

□ 価格設定の比較
  - 競合の価格帯
  - 顧客が感じる「値打ち」

成果物：`競合分析レポート`
```

#### 対応7-2: 差別化戦略（1～2週間）
```
■自社の「絶対的な強み」の定義と強調
  □ 機能面の強み（競合にはない機能）
  □ サービス面の強み（サポート品質 / 実装スピード）
  □ 産業知識 / ユーザー基盤
  
  □ マーケティング資料の見直し
    - 「競合と比較してここが違う」を明記

■顧客ロックイン戦略
  □ 既存顧客の満足度を高める
    - 定期的なフィードバック / 改善
    - 追加機能 / サービスの提供
  
  □ 長期契約 / 複数年契約への誘導
  □ ユーザーコミュニティ / アドボケイト育成

■市場ニッチの開拓
  □ 「大手競合がまだカバーしていない小さな市場」を狙う
  □ 業界別 / 企業規模別のセグメント戦略

成果物：`差別化戦略ドキュメント`
```

## リスク 8: 規制変更 / コンプライアンス違反

### リスク概要
- **事象**：個人情報保護法 / 消費者保護法などの規制が変わり、現在のサービス形態が違反状態に、または新たな許認可が必要になる
- **重要度**：Level 1～2（Critical / High）
- **発生確度**：低～中程度（業種・地域に依存）
- **影響度**：事業停止、罰金、社会的信用喪失

### 検知方法
```
■検知タイミング
  - 政府機関からの通達 / 法改正の発表
  - 業界団体からの情報提供
  - 法務専門家（顧問弁護士）からの警告

■検知指標
  - 新しい規制に対する「自社の適合度」が不明瞭
  - ユーザーデータ取得 / 利用に関する法的不確実性
```

### 対応フロー

#### 対応8-1: 法務確認（即座～5営業日）
```
チェック項目：
□ 顧問弁護士 / 法務コンサルタントに即座に相談
□ 「現在のサービス形態が新規制に適合しているか」確認
□ 必要な対応の内容・期限・費用を把握

成果物：`法務相談レコード・対応指示書`
```

#### 対応8-2: 対応計画の実行（期限までに）
```
■短期対応（数日～数週間）
  □ 利用規約 / プライバシーポリシーの修正
  □ データ取得 / 利用方法の変更
  □ ユーザーへの告知 / 新規約への同意取得

■中期対応（数週間～数ヶ月）
  □ システム・プロセスの修正
  □ データ管理の見直し
  □ 監査 / チェック体制の構築

成果物：`コンプライアンス対応計画書`、`修正済み規約等`
```

## 附録: リスク対応時の重要チェックリスト

### 全リスク共通：初期対応（最初の1時間）
```

```

  - 複数のリスクが同時発生した場合
  - 法的 / 社会的に大きな影響がある場合
```

  - 根本原因
  - 実施した対応フロー

  - 学んだ教訓
```
