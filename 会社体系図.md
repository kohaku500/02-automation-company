## 🔄 データフロー（詳細）

### Phase 1️⃣：戦略立案（09:00〜10:00）

```
COO判定部 (09:00)
  ├─ 経営指標の判定
  ├─ 各部署への指示を生成
  └─ 部署編成判定（追加・強化・廃止の推奨を記載）
      ↓ [推奨を参考に、次月の部署体制を検討]
マーケティング部 (10:00)
  └─ 10案の企画案を生成
      ↓
商品企画/企画案_YYYY-MM-DD.md
```

**COO判定の部署編成判定の基準：**
- **追加推奨の条件**
  - MRR が ¥30,000 超 → デザイン・品質向上部
  - 顧客満足度 4.0 未満 → カスタマーサポート強化部
  - コンテンツ生産量不足 → 第2コンテンツ制作部
- **強化推奨の条件**
  - 企画採用率が低い → マーケティング部のプロンプト強化
  - 品質スコアが低い → コンテンツ制作部の品質基準引き上げ
- **廃止推奨の条件**
  - 生産コストが高く効果が低い → 廃止検討
  - 重複機能がある → 統合検討

### Phase 2️⃣：コンテンツ生成（17:00）

```
商品企画/企画案_YYYY-MM-DD.md (マーケティング部の出力)
  ↓ [最売上予想案を選定]
コンテンツ制作部 (17:00)
  ├─ Gemini API 1回目: 投稿文・BOOTH・Kindle生成
  ├─ Gemini API 2回目: HTMLスライド生成
  ├─ Gemini API 3回目: HTMLアプリ生成
  │  （複数パスで品質・デザイン・UI/UX を最適化）
  ↓
商品パッケージ/YYYY-MM-DD/
  ├─ note_package.md (5,000文字以上)
  ├─ booth_package.md (7,000文字以上)
  ├─ kindle_manuscript.md (10,000文字以上)
  ├─ presentation.html (10〜15スライド)
  └─ app.html (完全自己完結型 HTML)
```

**コンテンツ制作部の「複数パス」の仕組み：**
- 1パス目：投稿文・各プラットフォーム向けコンテンツ生成
- 2パス目：HTMLスライドのデザイン・アニメーション強化
- 3パス目：HTMLアプリの UI/UX 改善・エラーハンドリング強化

### Phase 3️⃣：運営管理＆レポート（18:00）

```

運営ログ/日報_YYYY-MM-DD.md
```

## ✅ 各部署の責任

| 部署 | ミッション | 成功指標 | 出力 |
|------|----------|--------|------|
| **COO判定部** | 経営判断・部署編成 | MRR目標達成度 | 判定レポート + 部署編成推奨 |
| **マーケティング部** | 市場ニーズ把握・企画立案 | 採用率（売上実績）| 企画案×10本 |
| **コンテンツ制作部** | 高品質コンテンツの自動生成 | 品質スコア | 3プラットフォーム対応コンテンツ |
| **クライアント管理部** | 顧客満足度・リテンション | NPS / 返金率 | 顧客レポート |
| **秘書室** | 情報統合・次日指示 | 意思決定速度 | 日報・ToDo |

## 🎛️ 部署の動的編成

5部署体制は固定ではなく、**COO の判定に基づいて動的に編成される**。

### 部署追加フロー
```
COO判定で「追加推奨」
  ↓
秘書室の日報に記載・推奨を記録
  ↓
オーナーが必要性を確認
  ↓
GitHub で新ワークフロー ファイルを追加
  ↓
翌月より新部署が稼働
```
